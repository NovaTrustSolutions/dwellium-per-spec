- **`vi.useFakeTimers` + React 19 scheduler anti-pattern.** `vi.useFakeTimers()` mocks React 19's scheduler primitives (`MessageChannel` and microtask queue) that the React renderer uses for state-update commits in concurrent rendering mode. The `vi.useFakeTimers → vi.advanceTimersByTimeAsync → vi.useRealTimers` handoff is **empirically unsafe at jsdom + RTL + React 19 altitude** when the component-under-test has an async-cascade-to-render path: pending render commits stranded under fake primitives never fire on the real clock post-handoff. When the only Date-mocking need is `Date.now()` / `new Date()` anchoring (e.g., for "today" computation), use `vi.setSystemTime(...)` ALONE — it mocks the date without mocking scheduler primitives, leaving RTL's `waitFor()` to handle async settlement against the real clock deterministically. Reach for `vi.useFakeTimers()` ONLY when the component-under-test has explicit `setTimeout` / `setInterval` / `requestAnimationFrame` that the test needs to advance synchronously. Reference: `qualia-shell/src/test/appfolioParity/calendar.test.tsx` sister grid-regression (L212-255) + upcoming-events list (L257-294).

- **Per-provider-SSR-safety taxonomy (3-altitude classification).** Classify `localStorage` / `window` / `document` / `navigator` access patterns for SSR-safety verdict at each React Context Provider:
  1. **Initialization-time UNSAFE** — inside `useState(() => …)` lazy initializer OR `const X = localStorage.getItem(…)` at module top-level = fires during React render() = throws `ReferenceError: localStorage is not defined` on server-side render attempt.
  2. **Effect-time SAFE** — inside `useEffect(() => …, [])` OR `useLayoutEffect(() => …, [])` = fires AFTER hydration on client (no-op on server since useEffect doesn't fire during renderToString / renderToPipeableStream).
  3. **Event-handler-time SAFE** — inside callback functions (`onClick`, `onChange`, `setItem` mutators) = fires on user interaction post-hydration.
