## Deviations (all flagged in the original micro-plan; Ilya acked each)

1. **`RecurringCharge` over `TenantLedgerRow` / `InvoiceRow`.** The plan §7 line 290 names "`TenantLedgerRow` / `InvoiceRow`" as candidates. Chose `RecurringCharge` — maps to the literal seed filename (`recurring_charges.json`) and to the DoR's `monthly_charges` semantic (a recurring *schedule* vs. an invoice *event*). Tenant ledger vs. invoices are separate downstream concepts; either could be added later as its own interface without affecting this one.
2. **"7 optional fields" interpretation.** The plan's "7 optional fields" is loose phrasing. The 7 DoR-core fields are `account`, `amount`, `startDate`, `endDate`, `nextChargeDate`, `previousChargeDate`, `previousStatus`. The additional `paymentMethod?` is the "payment method enum" addition referenced by the plan title. Infrastructural fields (`id`, FKs, `notes`, timestamps) are Dwellium convention.
3. **3-rows-with-duplicate-annotation.** DoR's rows 2 and 3 are labeled "duplicate row (likely legacy charge-split or display artifact)". Preserved all 3 rows verbatim per Ilya's "Phase 1 is parity not cleanup" direction. Annotation carried in `notes`.
4. **`RecurringChargeStatus` non-nullable.** Ilya's nit — ack'd. Field typed as `RecurringChargeStatus | null` instead of folding null into the union.
5. **No permission key wired.** Added `'strata:accounting:recurring-charges'` to `TAB_PERMS` but did not seed the permission anywhere. Consistent with how other accounting tabs ship; permission-seed infra handles roll-out. God-role bypass covers local dev + CI.
