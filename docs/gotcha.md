# Gotcha — lessons from past Claude debugging sessions

Read this whenever a bug takes more than two turns to fix.
Add a tight entry whenever a Claude burns time on something avoidable.
Keep entries ≤4 lines so a full read stays cheap.

---

## Debugging discipline (apply on every project)

**Observe before patching.** If the first fix didn't work, the next thing you ship must be a log, not more code. Demand DevTools console output before writing fix #2. If the user can't or won't paste it, say so plainly — don't keep guessing into a void.

**Map every code path before debugging behavior.** A UI action often has multiple entry points (e.g. per-item button + bulk action). Bugs may live in only one. Ask exactly which control the user is clicking; read every handler for it side-by-side.

**One hypothesis, one change, then re-observe.** When uncertain, ship logs not features. Bundling speculative fixes piles up debt — you can't isolate which change caused the next failure.

**Trust authoritative signals.** When a docs lookup or focused subagent investigation has verified a pattern is correct, stop tweaking that area and look elsewhere.

**Renderer vs main-process logs** are in different consoles. `[Redline]`, `[acceptRedline]`, `[commentPlugin]` etc. only appear in the **Electron window's DevTools** (Cmd+Opt+I → Console). `[Honcho]` and main-process logs appear in the **terminal**. When asking for output, name the console.

## Holocron-specific priors

**CodeMirror 6 block widgets** must be provided through a `StateField` exposed via `EditorView.decorations.from(field)`. A `ViewPlugin`'s `decorations` spec silently drops them. Block widgets must have **no vertical margin** or you hit "Measure loop restarted more than 5 times" — wrap in an outer block with padding instead.

**Two redline-accept paths exist.** `acceptRedline` (per-card ✓ inside each yellow widget) and `RedlineNavigator.acceptAll` (bulk ✓ in the blue pill). Any per-redline side effect (resolving source comment, etc.) must be added to **both** or one path is silently broken.

**`lastRedlineSource` persists across turns** until consumed by `detectAndStoreRedlines` Path 1. The Address-all-comments flow must call `setLastRedlineSource(null)` before sending, otherwise stale Cmd+L state forces Path 1 instead of the multi-comment Path 2.

**Comment positions drift after redline accepts.** The `EditorView.updateListener` in `ScribePane.tsx` maps every redline `from`/`to` and every comment `fromLine`/`toLine` through `update.changes` so they stay aligned with the text after edits. Don't remove this — without it, accepting one redline corrupts the positions of all the others.

**Local model output is messy.** Gemma sometimes prefixes structured markers (`🍣 REDLINE:`, `**REDLINE:**`). Parsers must not anchor to start-of-line; match the marker pair shape and let leading decoration through.

**Block widgets must use `Decoration.replace({ block: true, widget })` from a StateField.** The table-rendering plugin (`tablePlugin.ts`) follows this pattern. ViewPlugin spec's `decorations` field silently drops block-replace decorations.

**Stale preload after preload changes.** When you edit `src/preload/index.ts` (new IPC bindings), the renderer's `window.electronAPI.X` stays as the OLD preload until **`npm run dev` restarts AND the renderer hard-reloads**. A symptom of this: `TypeError: window.electronAPI.X is not a function` even though you just added the binding. Fix: kill `npm run dev`, restart, Cmd+Shift+R the window.

**Chokidar must use `usePolling: true` on macOS.** fsevents through chokidar drops events for files added to pre-existing subdirectories — confirmed across multiple smoke tests in this project. `awaitWriteFinish` and `depth: undefined` alone don't fix it; only switching to polling does. Settled config in `workspace.ts`: `{ ignoreInitial: true, depth: undefined, usePolling: true, interval: 1000, awaitWriteFinish: { stabilityThreshold: 250, pollInterval: 50 } }`. **Don't switch back to fsevents** — every "let's see if fsevents works now" attempt in this repo's history has lost time. Polling at 1s is slower but deterministic, and CPU cost on a single-user project tree is negligible.

**There is exactly ONE workspace watcher.** `startWatcher` in `workspace.ts` does `if (watcher) { watcher.close() }` then replaces — calling it a second time KILLS the previous watcher. So `startWatcher(threadPath)` after the root has been started silently disables ingestion for every other thread under the root. Thread switching and thread branching MUST NOT call `startWatcher`. The root watcher covers all thread folders recursively; thread activation is a renderer-state concern. This bit us on 2026-05-08 — `[Workspace] watching: …/Holocron Build` followed every thread switch and broke wiki-compile smoke testing.

## v13 priors — three-tier wiki, graph, boot self-healing, Codex link resolution

**`npm run typecheck`'s `&&` short-circuits.** The script is `tsc --noEmit -p tsconfig.node.json && tsc --noEmit -p tsconfig.web.json` — if the node side has *any* error (and there are pre-existing ones in the handoff table), the web tsc never runs and renderer errors stay invisible. To verify a renderer change: `npx tsc --noEmit -p tsconfig.web.json` directly. Also: `npx tsc --noEmit` from the editor root is a no-op (root tsconfig is `"files": []` + project refs); `tsc -b` emits `.js`/`.d.ts` into the source tree. Use the per-project no-emit invocations only.

**A migration that touches `rag_wiki_pages` must also clean `rag_documents`.** `reingestAsWiki` creates a `rag_documents` row (`source_type='wiki'`, `source_path = _Codex/Wiki/<slug>.md` — `_Library` pre-v15) for every compiled wiki page. Deleting/wiping `rag_wiki_pages` without deleting those doc rows leaves zombies — orphaned doc rows + their orphaned `rag_tags` (the `rag_document_tags` links cascade, the tag *definitions* don't). Migration 007 did exactly this → 143 zombie wiki docs + ~314 orphan tags. The repair is `deleteZombieWikiDocs()` (`cleanupOps.ts`), now run on every boot.

**Wiki orphan-sweep symmetry: `deleteSourcelessWikiPages` and `scanOrphans` must agree.** They use the same three-tier "sourceless" definition — thread tier = no `rag_wiki_page_sources` rows; project tier = no surviving thread-tier sibling in the same `(namespace, domaine_id)`; domaine tier = no surviving project-tier sibling in the same `domaine_id`. If the read-only count uses one rule and the destructive sweep uses another you get the "badge says N, sweep deletes 0" mismatch (this happened with the v007 refactor: tier-2/3 pages never have `rag_wiki_page_sources` by design, so the old "no source rows = orphan" rule flagged every healthy tier-2/3 page).

**Boot-time self-healing is the maintenance-agent pattern.** Every structural invariant gets checked and repaired on app start, idempotently: `syncWorkspaceRoots` (config-key drift), `validateActiveConfigPaths` (stale active thread/project), `bootstrapMissingPages` (uncompiled wiki tiers), `deleteZombieWikiDocs` (migration residue), and (manual today, should be boot too) `sweepOrphans`. Each logs unconditionally (`[Boot] …`) so a no-op pass is still visible in dev. New invariants should follow this pattern, not require manual intervention.

**Three workspace-root config keys drift: `holocronRoot` / `projectsRoot` / `workspace.path`.** They've each historically been written by a different surface (Settings → Connections, the Domaines folder picker, a legacy fallback) and silently diverged — causing "create Domaine doesn't make a folder" and `..`-prefixed path rejections. `syncWorkspaceRoots(cfg, sourceOfTruth?)` in `config.ts` keeps all three in lockstep; it runs on boot to heal existing configs, and both folder pickers now write all three. Always go through it.

**Codex Domaine filters live in their stores, never in component state tied to `activeDomaineId`.** Every Codex sub-tab's Domaine selector (`ingestStore.selectorDomaineId`, `codexWikiStore.selectorDomaineId`, `graphStore.selectorDomaineId`) defaults to `''` = "(All Domaines)" and persists across tab switches. The original bug pattern — `useState(config.activeDomaineId)` + a `useEffect(() => setSel(activeDomaineId), [activeDomaineId])` — reset the filter to "AI" on every mount. If you add a filter that should persist a session, it goes in the store with a default that isn't bound to active context.

**`[[wikilinks]]` in raw documents resolve through `openDocByName`, not just the wiki page index.** A reference doc's `[[SR 9 - Master Report - …]]` points at *another `rag_documents` row*, not a `rag_wiki_pages` page. `CodexPreview.handleWikilinkClick` → `openDocByName(name)` which tries the wiki index first, then `ingestListDocuments({ search: name, crossDomaine: true })` matched by filename (Obsidian `[[name]]` ≡ `name.md`), then DB title, then substring. If nothing matches it `setError("No document or wiki page matches …")` — **that's not a bug, it means the target `.md` isn't ingested yet**; the fix is to run Sync workspace, not touch code.

**In the Electron renderer, a bare `<a href>` does nothing.** `webContents.on('will-navigate', e => e.preventDefault())` in `main/index.ts` blocks in-window navigation. External links must go through `window.open(href, '_blank', 'noopener,noreferrer')` — the main process's `setWindowOpenHandler` hands that to `shell.openExternal` (system browser / mail client). And `react-markdown` 8.x sanitizes link URLs by default — `transformLinkUri={(uri) => uri}` (identity) is required so custom-protocol hrefs (`holocron-wiki:`, `wiki://`) reach `components.a` unstripped. In `CodexPreview`'s `MarkdownReader`, **no `a:` branch may render a click-does-nothing `<a>`** — every branch attaches a handler (`#anchor` → explicit `scrollIntoView`, empty href → inert, everything else → `onInternalLinkClick`).

**fcose, not cola, for clustered graphs (and d3-force superseded both).** `cytoscape-cola`'s continuous physics fights compound-node clustering; `cytoscape-fcose` is compound-aware (packs children inside parents, tiles disconnected components). The current renderer is `d3-force` (Cytoscape removed entirely, −1.3 MB bundle, +270 KB d3): continuous "living cell" feel = `simulation.alphaMin(0).alphaTarget(0.012)` (never freezes, perpetual micro-drift) + `velocityDecay(0.42)` (friction without overdamping); Domaine clustering = `forceX`/`forceY` toward per-Domaine anchor points on a ring (no bounding boxes), not compound nodes. Don't reach for Cytoscape again.

**`detectSourceType` must use path-position logic, not folder-name regex.** The gate is "is this file inside a thread?" (path segments 0/1/2 = Domaine/Project/Thread), not "is this folder named References/?". Any `.md` under `<projectsRoot>/<D>/<P>/<T>/<anything>` ingests with `source_type` inferred from filename/folder hints (`BD_*` → brain_dump, `Notes_*` / `Notes/` → note, `Reports/` → report, `References/` / `Transcripts/` → reference), defaulting to `'reference'`. **Never reject based on unknown folder names** — `Drafts/`, `Meetings/`, anything Andy creates just works. The prior regex-per-known-folder approach silently rejected user-created subfolders and is gone (`22f651e`).

## v14 priors — analytics, `_Codex` rename, graph polish, rename-recompile

**Vite caches aggressively — flush `.vite` cache if hot reload feels stale.** If a `console.log` you added doesn't appear, or your code change produces no DevTools effect after a save+reload, suspect Vite's persistent dep / module cache before suspecting your code. Fix: `rm -rf node_modules/.vite && npm run dev`. Verify the chunk hash actually changed (DevTools → Sources → look at the loaded JS filename — it should differ after a real cache miss). This bit us when a `[GraphClick]` diagnostic added to the click handler showed no output for several restarts despite the file being saved correctly.

**Click-to-zoom on graph nodes with d3-force + dynamic container width is hard — deferred to v4 Session 5.** Multiple attempts in v14 (zoom to k=2.5, smooth pan+zoom, sim cooling + node pinning, etc.) all ran into the same problem: the detail rail mounts ON selection, stealing ~306px from the graph container width AFTER the click handler has already computed the target transform. The first click of each session lands ~150px off-center; every subsequent click is correct because `getBoundingClientRect()` reads post-rail dims. Workarounds tried + rejected: subtract a hardcoded `RAIL_WIDTH = 306` (over-compensates in collapsed mode), add `railCollapsedRef` + `railWidthRef` mirrors (scope creep), `requestAnimationFrame` to defer the transition (laggy feel). Current behavior shipped in v14: **pan-only** at current zoom (200ms easeCubicOut, no scale change, no sim cooling, no rail compensation). Click-to-zoom revisits in Session 5 with proper isolation — likely solution is `flushSync` on the selection state so the rail mounts before the transition's target is computed.

## Session 2 priors — Honcho memory, context measurement, UX consolidation

**Context window was hardcoded to 8,192 regardless of model.** The chat-bar reading used `ai.contextWindow` (LM Studio's setting, default 8 K) for every provider — so Gemini Flash (1 M tokens) and Claude Sonnet (200 K) showed bars at 90 %+ just from opening a doc. Fixed in `ChatPane.tsx` by routing through `activeProvider`: `gemini → gemini.contextWindow (1,048,576)`, `anthropic → anthropic.contextWindow (200,000)`, `lmstudio → ai.contextWindow (8,192)`. If the bar reads wrong, check `activeProvider` in the destructure (`const { …, activeProvider } = config`) and the per-provider `contextWindow` field in `settingsStore` defaults. **The per-provider fields are user-editable in Settings → Connections** so Andy can also tune them — e.g. if he ever uses Opus 4.7's 1 M tier.

**Memory controls live in TWO places by design: chat header + Settings → Maintenance.** The chat header is intentionally minimal — two buttons only: `⟲ Reset` (orange, the persistent compress-and-rotate) and `Memory ▸` (cyan, opens the inspection drawer). **Do not add more controls to the chat header.** Inspection-only affordances go inside the Memory panel (session id, duration, dream insights, Dream-now, Memory file viewer with synthesisReady badge, footer deep-link to Maintenance). Destructive actions go in Settings → Maintenance as an A→B→C escalation: amber-bordered "Clear thread memory" (type the thread name), pink-bordered "Clear all thread memory" (type CLEAR ALL), red-bordered "Nuclear Reset" (type RESET). The deep-link from MemoryPanel uses `useSettingsStore.openSettingsAt('maintenance')` — that's the navigation pattern; copy it for any future tab deep-links.

**Eagerly loading all session summaries into the system prompt defeats compression.** Pre-fix `buildSystemMessage` stamped the 5 most-recent summaries verbatim into every request → context at 62 % post-compression. Post-fix: at most ONE brief paragraph (`trimSummaryForPrompt` caps at 400 chars / first paragraph / sentence boundary) plus a one-line pointer to the resolved `Memory_<Project>_<Thread>.json` path. Older summaries live on disk only — the agent cites the file by path when the user asks about prior sessions. **If you find yourself reaching for `sessionSummaries.map(...)` again, you've forgotten this lesson** — Honcho is designed to hold memory server-side; the system prompt is for the brief orienting hook.

**`honchoDream()` degrades silently — never break the chat flow.** The Dialectic `/chat` endpoint may not be deployed in older Honcho instances. Any non-2xx → `{ ok: false, error }` + a `[Honcho/Dream] ← N (degrading)` log line. The Memory panel surfaces "No insights yet" honestly. **Don't add error toasts or modal dialogs around dream failures** — they're expected on some deployments and have no user remediation. Same pattern applies to `deleteSession` (DELETE on session may 404 / 405) — falls back to "rotate locally, abandon old session" and the success message says so honestly.

**Compress button can succeed without the bar moving.** When the active doc is the dominant load (large file on a small-window LM Studio model), Compress rotates the Honcho session but the bar stays high because file content is unchanged. Two UX measures address this: (1) `setContextWarningDismissed(true)` post-success so the prompt doesn't immediately re-fire (re-arms naturally on bar drop), (2) a "Compressed at HH:MM" 5-second in-chat toast so the action is visible. If you find yourself debugging "Compress doesn't work", first confirm `[ChatPane] reset context complete: …` lands in the console — if it does, the button worked; the bar staying high is honest (file is the load), and the diagnostic is to truncate file content (Option A in the original diagnostic — deferred).

**`synthesisReady` is sticky-true, never falls back.** Once `summaries.length >= 3 || dreamInsights.length >= 3`, the flag flips to `true` inside `appendMemorySummary` + `appendDreamInsight` and stays there for the life of the Memory file. **Do not recompute it on read or invert the rule** — the Hive Session 3 consumer assumes once-flipped-always-flipped. If a thread shrinks its dreamInsights[] manually (which shouldn't happen — they're append-only), the flag stays true. That's intentional.

**Honcho v3 memory works via Conclusions — logical facts derived from interactions.** CoPaw (Session 3) should POST extracted facts to `/v3/workspaces/holocron/conclusions` as Conclusion objects (`content`, `observer_id='andy'`, `observed_id='andy'`), not rely on Honcho to auto-derive them from raw messages. The dialectic/chat endpoint synthesizes FROM conclusions — so the more conclusions we write, the better the dream output. **Schedule Dream endpoint:** `POST /v3/workspaces/{workspace_id}/schedule_dream` with `{ observer: 'andy', dream_type: 'omni' }` — use this instead of the chat endpoint for triggering dreams. **Session context:** `GET /v3/workspaces/{workspace_id}/sessions/{session_id}/context` returns messages + summary in one token-limited call — use for Reset Context instead of separate calls (currently `thread:reset-context` does `getSessionContext` then a separate `getMessages` on fallback; the single-call shape is preferable).

## Session 3 priors — CoPaw, Dreams panel, Synthesis Agent, schedule_dream

**CoPaw is silent by design — no UI.** Architecture-v4 §4.6 spec'd this as "the compounding loop that makes Honcho smarter over time *without* explicit memory management." Output is visible only in DevTools `[CoPaw] extracted N conclusion(s) from response (X chars)` and (eventually) via the Honcho Dreams panel and synthesis quality. **Don't add UI for it** unless heuristic noise becomes a measured problem — adding a panel undermines the "compound silently" intent. If extraction becomes too noisy/sparse to live with, the v2 upgrade is a Gemini-Flash "pull the 3-5 durable facts from this" pass, NOT a manual-curation UI. Implementation lives in `src/renderer/src/components/chat/useLMStream.ts:fireCoPaw()` — fires in the `onComplete` branch of the LM stream's end handler, after the assistant message lands in chat AND after `honchoSaveMessage` runs. Skips redline responses (editor-targeted, not durable facts) and silent flows (Address-all preset).

**Dream action state (rejected / deferred) is in-memory only — resets on restart.** `hiveStore` holds `rejectedDreamIds: Set<string>` and `deferredDreamIds: Set<string>`. Approved dreams are effectively absorbed into a synthesis document and stay out of the panel because they're removed from the visible list (we mark them rejected after Approve succeeds); the underlying `dreamInsight` entry in the Memory file stays put forever. **To make Reject/Defer durable across restarts**, add a `dismissedAt: string | null` field to the dreamInsight entry shape in `projectFs.ts:MemoryFile`, expose a `thread:dismiss-dream-insight` IPC, and have `markDreamRejected`/`markDreamDeferred` write through to disk. Session 3 deliberately deferred this — Andy hadn't asked for it yet and the in-memory store was simpler to ship.

**Synthesis Agent writes to `_Codex/Syntheses/<slug>.md` and inserts a `rag_syntheses` row.** Chokidar (`workspace.ts` polling watcher) picks up the new file automatically — no manual re-ingest needed. The doc enters the Codex naturally through the existing `add`-event ingestion pipeline in `ragIngest.ts`. **If the doc fails to appear** in Codex → Ingest after a synthesis generation, the diagnostic is `[Workspace] watching:` log line in the terminal — confirm chokidar is watching the workspace root (not just the active thread folder — see the `gotcha.md` "There is exactly ONE workspace watcher" entry from v13). The Synthesis Agent's `disk_path` field on the `rag_syntheses` row points at the file; the loop-back rag_documents row gets created by the standard ingestion path, NOT by the synthesis writer.

**`schedule_dream` returns 204 (no body) on success — don't try to parse a response.** `honcho.ts:scheduleDream()` reads `res.status === 204 || res.ok` and returns `{ ok: true, status }` with no content. The `honchoDream()` wrapper treats success as `mode: 'scheduled'` and short-circuits — no `/chat` fallback. The fallback fires ONLY when schedule_dream returns non-204 (typically 404/405 on deployments without the endpoint). **If you see `[Honcho/Dream] scheduled via schedule_dream (no synchronous insight)`** that's the happy path on a modern Honcho — the dream is running server-side and Holocron has no synchronous insight to render. The Memory panel surfaces this as `Dream scheduled — Honcho will process this in the background.` honestly. Don't add code that expects a synchronous insight in the scheduled branch — there isn't one to read.

**Synthesis documents should have `source_type='synthesis'` not `'reference'`.** Files written by the Synthesis Agent land at `_Codex/Syntheses/<slug>.md` and get picked up by chokidar as a normal `add` event → `ragIngest.detectSourceType()` classifies them. Currently `detectSourceType` has no path-position branch for `_Codex/Syntheses/` so synthesis docs ingest with whatever the default is (typically `'reference'`) — they're queryable but mistagged. **Fix in Session 4**: add a path check inside `detectSourceType` (mirrors the existing `_Codex/Wiki/` → `'wiki'` branch and the v13 path-position-not-folder-name rule). Once classified correctly: graph badges, source-type filters in Codex Search, and the Syntheses analytics layer's own self-counting all become honest. The companion fix is to ensure the `rag_syntheses.disk_path` ↔ `rag_documents.source_path` invariant matches the wiki pattern so the same boot self-healing (`deleteZombieWikiDocs` shape) applies to syntheses too.

**Global Hive chat (cross-Codex RAG, not thread-scoped) is a Session 7 addition.** The Hive currently surfaces per-agent monitoring cards (Honcho / Synthesis / Validation / Ingestion) but doesn't have its own chat affordance — Andy's chat surface stays in Scribe and is scoped to the active thread's Honcho session. **For open-ended questions that aren't thread-specific** (cross-Codex retrieval, "what do I know about X across everything", debugging-the-knowledge-base questions), the workaround until Session 7 is a dedicated `General` thread in Scribe with a generous system prompt and no RAG scoping. Don't add a chat panel to the Hive tab in Sessions 4–6 — the Working Memory panel work (Session 7) is the natural carrier for the "grounded in" instrumentation that makes a global-RAG chat trustworthy, and the two should ship together.

## Session 4 priors — Foundry, Triage Agent, intake modes

**Foundry triage logs are in the TERMINAL, not DevTools.** `src/main/foundry.ts` runs in the main process — every `[Foundry/triage] starting for item …` and `[Foundry/triage] item … triaged (mode=…, … cleaned=… chars)` lands in the `npm run dev` terminal output, NOT in the renderer's DevTools console. Same gotcha pattern as `[Honcho]`, `[Workspace]`, `[Boot]` (line 19 of this file). When debugging a "triage didn't fire" case, watch the terminal first. The entry-point log was added in Session 4 specifically because the next debugging round will resolve in one step rather than the hour-long DevTools red herring that prompted it. If you don't see `starting for item`, the function isn't being called (preload binding, capture-flow bug); if you see `starting` but not `triaged`, the failure is inside (look for `attempt N parse failed` / `attempt N chat error` / `triage failed after 2 attempts`).

**Cookie boilerplate is stripped before the 8000-char triage window.** `stripCookieBoilerplate()` in `foundry.ts` runs URL-only (paste/file are user-authored). Algorithm: lowercased `lastIndexOf` for each phrase in `['we value your privacy', 'customize consent', 'accept all', 'cookieyes']` → take max end-position across all phrases → find next `\n## ` or `\n# ` heading → slice from there. If stripping fires you'll see `[Foundry/triage] skipped N chars of boilerplate before first article heading` in the terminal. **The reason this is needed**: many sites lead with a consent modal that Firecrawl captures verbatim, and if those ~2-3 KB of cookie text fill the opening of the 8 KB prompt window, Gemini ends up classifying + cleaning the banner instead of the article. The heuristic is intentionally narrow (4 phrases, only triggers when the LAST hit is followed by a heading). To add a new cookie family, lowercase the marker phrase and append to `COOKIE_PHRASES`. Don't move the call site — it MUST run before the per-source content cap slicing, otherwise the cap cuts the article body instead of the boilerplate.

**Convert-only mode skips `cleaned_content`, uses 1024 maxTokens (vs 8192 for extract), completes in 1-2s.** `foundry_items.triage_mode` (migration 011) drives which prompt Gemini gets: `'extract'` includes the cleaning rewrite instruction; `'convert'` omits it entirely and only asks for the 4 metadata fields (tags/domain/score/signal). The convert-mode UPDATE forces `cleaned_content = NULL` as a safety belt even if Gemini went off-script and emitted one. **Use Convert for: contracts, verbatim transcripts, source documents, code/config files Andy needs preserved word-for-word.** Use Extract for noisy web content where the article body is buried in nav/ads/boilerplate. Defaults: URL = extract, file = extract, paste = convert (paste is hard-coded in the renderer — the toggle isn't shown there because user-authored content should never be summarized). The Approve flow follows: extract mode writes `cleaned_content` to disk; convert mode writes `raw_content`. Both paths go through the same `_Codex/References/` or `<thread>/References/` admission logic and chokidar pick-up.

**Two Foundry modes by design: Extract signal vs Convert only.** This isn't a transient UX experiment — it's a load-bearing distinction that should NOT be unified into a single "smart" path. The user is the quality gate for what enters the Codex (architecture-v4 §6.5), and the agent doesn't know whether Andy wants his contract paraphrased or preserved verbatim. The toggle lives on the URL + file panels at capture time so the choice is made before triage runs (changing it post-capture would require re-triaging, which we don't support yet — see HANDOFF_v16 known-limit §5). The triage chip in the Review queue surfaces the mode honestly: `CLEANED BY TRIAGE` (green, extract + cleaned populated) vs `PRESERVED` (cyan, convert mode) vs `TRIAGED` (cyan, extract but Gemini omitted cleaning) vs `TRIAGE PENDING` (orange). If you ever feel tempted to "auto-detect" the mode (e.g. "short content = convert"), don't — the explicit choice is honest about the trade-off and Andy has stated this preference.

## Session 5 priors — Hermes, iCloud watcher, PDF/DOCX extraction

**Hermes long-poll uses a dedicated fetch with 30 s Telegram timeout + 60 s ceiling + exponential backoff.** `getUpdates?timeout=30` is Telegram's long-poll convention; the outer `fetch` adds a 60 s `AbortSignal.timeout` as belt-and-suspenders for hung Cloudflare proxies. Errors back off 5 s → 60 s ceiling, reset on success. **Don't share the abort controller with other fetches** — `pollAbortController` exists exclusively to let `stopHermesBot()` cleanly cancel an in-flight long-poll AND any pending `sleep()` backoff. If you wire a new fetch through the same controller, calling stop will kill that fetch too. Use `AbortSignal.timeout(...)` for per-call timeouts; reserve the module controller for the loop lifecycle.

**iCloud watcher is a SEPARATE chokidar instance from `workspace.ts:startWatcher`.** Never merge them — gotcha line 41 lesson. `startWatcher` kills its previous watcher on every call, so reusing it would silently disable Codex ingestion the moment the iCloud watcher starts. The two watchers MUST stay independent: workspace watches `cfg.projectsRoot` (the `_Domaines/` tree), iCloud watches `cfg.icloudInboxPath`. Same polling config (`usePolling: true`, 1 s interval, awaitWriteFinish 250 ms — gotcha line 39 — never fsevents on macOS), but separate `chokidar.watch(...)` calls and separate `FSWatcher` instances at module scope in `hermes.ts`. If you find yourself reaching for a unified "start all watchers" helper, stop — explicit-and-separate is the safer shape.

**Telegram `/dream` reply is honest about being async.** `schedule_dream` returns 204 (no body) — there's no synchronous insight to surface back via Telegram. The bot replies `🌀 Dream scheduled. Honcho is synthesizing in the background — check the Hive Dreams panel later.` Don't add code that expects a synchronous insight in the scheduled branch — there isn't one to read. This matches the gotcha.md Session 3 prior on `schedule_dream` behaviour; the Telegram surface is now another caller that must respect the same contract. If a Hermes user complains "I asked for a dream and got nothing in chat" — that's the honest state; the insight lands in the Hive Dreams panel when Honcho's deriver completes server-side.

**Bot display name is "Hermes" — internal Honcho workspace id stays `holocron`.** The user-facing relay agent is named Hermes (per architecture-v4 §4.5), but the Honcho workspace constant in `honcho.ts:1` (`const WORKSPACE_ID = 'holocron'`) MUST remain `holocron`. Renaming it would orphan every existing thread's `honchoSessionId` because sessions live under workspace-scoped paths (`/v3/workspaces/holocron/sessions/...`). Don't rename. ## Session 8 priors — UI Overhaul lessons learned

**`position: sticky; top: 0` pins relative to the scroll container's PADDING BOX — move chrome padding to an inner wrapper or the sticky element won't reach the visual top.** Pre-Part-C, `chatScrollRef` had `className="flex-1 overflow-y-auto px-3 py-3"`. With `py-3` on the scroll container, a sticky child with `top: 0` pins 12 px below the visual top edge of the chat viewport because sticky positions relative to the padding box, not the content viewport. Fix: move ALL padding off the scroll container (`padding: 0`) and onto an inner wrapper div that holds the message list (`<div className="px-3 py-3">`). The sticky child now reaches the visual top edge cleanly. Negative margins on the sticky child (e.g., `-12 px ml/mr` to bleed the user-message band to the chat panel edges) still escape the inner wrapper's padding correctly. **This applies to every `position: sticky; top: 0` pattern in the codebase, not just the chat.** If you find a sticky element that "almost" pins at the top but has a few-pixel gap, check the scroll-container padding first.

**Every `ChatMessage.tsx` markdown handler must use theme-aware tokens — `var(--text-1)`, `var(--text-4)`, `var(--neon-blue)`, or `color-mix` expressions over `var(--bg-2)` / `var(--text-1)`. Never hardcode `#ffffff`, `rgba(255,255,255,...)`, or other dark-only literals.** Theme 5 in `themes.ts` (line 125) is the light theme: `bg-2 #eeeeee`, `text-1 #111111`. Hardcoded dark-only colors fail catastrophically on it: a `<strong style={{ color: '#ffffff' }}>` is invisible on a `#eeeeee` background; a `<code style={{ background: '#1c1c1e' }}>` produces dark-text-on-dark-block (text-1 = `#111` inherits, sits on `#1c` bg) = unreadable. The Session 8 Part C audit replaced six handlers: `strong` color dropped (inherits text-1; bold weight alone provides emphasis per markdown convention); inline+block `code` backgrounds via `color-mix(in srgb, transparent 88%, var(--text-1) 12%)` and `color-mix(in srgb, var(--bg-2) 80%, var(--text-1) 20%)` respectively (theme-aware contrast in both directions — lighter on dark, darker on light); `blockquote` border `#3a3a3c` → `var(--text-4)`; non-holocron link `#0a84ff` → `var(--neon-blue)` (per-theme accent); thinking-placeholder span `rgba(235,235,245,0.3)` → `var(--text-4)`. **Before adding any new markdown handler, audit it against the light theme in your head: does it use var(--text-1) / inherit / var(--neon-blue) / color-mix? If not, you've shipped a light-theme bug.** Same rule applies to any other component that renders user-content text — Foundry's ReviewPanel, CodexPreview's MarkdownReader, etc. If you find yourself typing `'#ffffff'` or `'white'`, stop.

## Session 9 priors — Code Health pass

**`withRagClient<T>` returns `Promise<T | null>`, not `Promise<T>`.** The pool can fail to open a client (DB down at boot, transient pg error) and the function returns `null` rather than throwing. Every caller must coalesce: `const n = (await withRagClient(...)) ?? defaultValue`. The Session 9 fix at `cleanupOps.ts:508` exposed this — `deleted` was typed `number | null` and the function return type required `number`. Don't shave the `??` off thinking the inner `return res.rowCount ?? 0` already coalesces — that's the *callback's* return; the outer pool guard is a separate layer.

**`pg.QueryResult.rowCount` is `number | null`** under modern `@types/pg`. Older code wrote `res.rowCount > 0` and TS only started flagging it when `strictNullChecks` propagated through the call graph. Use `(res?.rowCount ?? 0) > 0` everywhere — both the result and the rowCount can be nullish. The Session 9 fixes at `dashboard.ts:54` and `cleanupOps.ts` use this pattern; copy it forward.

**`.map(basename)` — `basename`'s second param (suffix:string) collides with `.map`'s second callback arg (index:number).** TS catches this via the strict-overload-pick. Fix is `.map((p) => basename(p))` — explicit arrow drops the extra args. This is the same shape as `arr.map(parseInt)` infamously producing `[1, NaN, NaN]`; any unary util whose API exposes a second optional parameter should be wrapped in an arrow when passed to `.map`. See Session 9's `ipc.ts:527` fix for the canonical example.

**`react-markdown`'s `components` prop only accepts valid HTML tag names + a small fixed set of element keys. Slots like `p_last` are silently ignored at runtime AND flagged by TS2353.** The Session 9 cleanup removed a `p_last: ...` handler from `ChatMessage.tsx` that had been in place since Session 8 Part C; it never did anything (the library has no such hook). To style "the last paragraph differently" you need a different mechanism — a `:last-child` selector on the outer `<p>` styles, a custom plugin that swaps the tag, or post-process the rendered tree. Don't reinvent the `p_last` hack.

**`ViewPlugin<unknown>` is rejected by `@codemirror/view`'s generic constraint — the type parameter must extend `PluginValue`.** `PluginValue` is just an interface with optional `update?(u)` and `destroy?()` — any plugin class trivially satisfies it. Use `import { type PluginValue } from '@codemirror/view'` and `ViewPlugin<PluginValue>` (or just let inference do it). The Session 9 fix at `selectionObserver.ts:14` was a 2-line change.

**Renderer-side `editorTheme.customs` types its tokens loosely as `Record<string, string>`; the canonical `ScribeTokens` (in `scribeThemes.ts`) is `Record<ScribeTokenKey, string>` (named keys: `bold`, `link`, `h1`, `h2`, …).** Three call sites (`App.tsx`, `ScribeTab.tsx`, `CodexPreview.tsx` post-Session-9) cast `customs as Record<string, ScribeColorTheme>` to satisfy `resolveTheme(name, customs)`. The "proper" fix is to tighten `customs` in `settingsStore.ts` itself, but that risks hydration breakage if a persisted config has missing token keys. The Session 9 decision was to mirror the existing cast pattern at the third site rather than widen-then-heal. If a fourth site appears, do the same; if the cast count ever exceeds five, revisit the store typing with a hydration migration.

**`config.gemini` / `config.anthropic` are now declared on the main-side `HolocronConfig` (Session 9 Part C). Don't add `(cfg as unknown as { gemini?: { apiKey?: string } }).gemini` workarounds — they're gone.** Five call sites used the pattern (ragIngest, ragWiki, foundry, ipc, hermes) plus a sixth local-intersection in dashboard.ts; all collapsed to `cfg.gemini.apiKey` / `cfg.anthropic.apiKey` direct reads. Rule going forward: if a new code path needs a config field, add it to BOTH `src/main/config.ts` HolocronConfig (with a default in DEFAULT_CONFIG) AND `src/renderer/src/store/settingsStore.ts` HolocronConfig (mirror the shape) so the renderer + main agree. If you find yourself typing `cfg as unknown as { … }`, you've forgotten the lesson.

**Filesystem-side test harness is at `tests/fs-harness.ts` (Session 9 Part D). Real chokidar is deliberately NOT invoked.** Reasons documented in the file's header: polling adds 1–2 s per test; the workspace.ts watcher is module-scoped (one watcher mutated by every startWatcher() — see gotcha.md line 41) and would bleed across test files; macOS-fsevents concerns are irrelevant under vitest. Pattern for future fs-side tests: stand up a `setupWorkspaceTree()`, mutate fs + DB in lockstep (the harness exposes `setDocActiveByPath` + `isDocActiveAtPath` mirroring the guard's SQL shape), then assert against the conditions the production code checks. The wikilink-writeback-guard smoke test (`tests/wikilink-writeback-guard.test.ts`) is the canonical example — 5 cases covering happy path + four failure modes (file deleted, row soft-deleted, both, move-to-thread).

## Session 10 priors — Domain color editing + active-thread rename

**Active-thread rename is safe under `withRenameLock` + the Session-7 wikilink-writeback guards. The historical `assertNotActiveThread` guard at `renameThread` was paranoia, NOT load-bearing.** Session 10 removed it (projectFs.ts:1174 pre-change). ## Architecture priors (read before Phase 3 work)

**Carpathia "magic" RAG is Level 4, not Level 5/6.** The TikTok creator who cried built an Obsidian-style structured wiki + tags + agent-curated relationships, NOT vector RAG. Holocron's Phase 3 spec (`architecture-v3.md`) builds Level 4 first; defer vector embeddings (Level 5) to Phase 3.5 if/when keyword + tag retrieval hits a wall.

**iCloud Drive is the cloud transport, not Google Drive API.** Holocron's persistent state is markdown files in an iCloud-synced root. Files are accessible cross-device via the Files app (iOS/macOS) for inspection, but Holocron itself is the only app that ingests, indexes, and rewrites them. Don't write OAuth or Drive API code — the OS handles sync.

**Wikilinks are a free relationship layer (Holocron-native).** When the ingestion agent inserts `[[Page Name]]` into documents, Holocron's editor renders them as clickable links AND stores them as edges in `rag_relationships`. The format is a markdown convention, not a vendor-specific syntax — files remain plain markdown if exported.

**Provider routing by task, not by mode.** Gemini Flash for structural transformation (tagging, wikilinking, wiki compilation, retrieval). Claude Sonnet for synthesis essays + report generation. Hot-swap pill in chat for manual override. Never put synthesis on Flash; never put bulk ingestion on Sonnet.

**Database is index, files are truth.** Postgres + tsvector indexes the file content. If the DB gets corrupted, re-ingest from disk. Never store data only in DB without a corresponding file artifact (synthesis essays write to `_Codex/Syntheses/*.md`, wiki pages write to `_Codex/Wiki/*.md` — both renamed from `_Library/*` in v15).
