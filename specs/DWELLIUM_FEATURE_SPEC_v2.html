02Current Build — Observations

Observed under the port-conflict environment. Items tagged "Verify Friday" should be retested after a clean install.

2.1 — Theme and Color System
Verify Friday

Theme switching did not apply uniformly in the version reviewed — some panels updated in light mode while others did not. Some elements appeared to have color values that did not respond to theme changes. Ilya indicated colors are configurable in Settings; this could not be confirmed remotely. When attempting to change colors, the result was not as expected. Will verify together in person.

Desired outcome regardless: all colors in all components should reference CSS custom properties so a new theme requires only updating variable values, never touching component stylesheets.

2.2 — Module Connectivity and Error States
Verify Friday

Several modules showed error states or unconnected placeholders. Some of this is likely the port conflict. Modules that are genuinely not yet connected should show a clean “not yet connected” placeholder rather than a React error. The Inbox widget opened too small to be usable.

2.3 — API Connectivity
Verify Friday — will configure together

After entering an Anthropic API key and switching the active provider to ARA, ARA reported an error and did not respond. Will be resolved in person. Going forward: active provider clearly visible in settings, live connection status per provider, “Test” button for each.

2.4 — Workspace and File Location
Observation

When files are created or saved, there is no visible indication of where they are stored. Workspace root path should be visible in the interface; saved files should be navigable from the file explorer.

2.5 — System-Wide Content Search
Feature Needed

A content search — finding documents, notes, or any data by keyword or natural language — does not appear to be present. The sidebar filter is navigation, not search. If vector/semantic search has been built into the backend, it needs a prominent query interface.

2.6 — Version Button Does Not Increment
Bug

The version button in Scribe repeatedly saves as “version one” without incrementing. Correct behavior: each click creates a new numbered file (v1 → v2 → v3), preserving all prior versions. The reference implementation does this correctly.

2.7 — Chrome / Content Ratio
Design Issue

Multiple windows devote too much height to title bar, banners, and navigation rows, leaving too little room for actual content. A professional rule of thumb: content should occupy at least 70% of the available height.

  • Remove the persistent “Drag outside window to pop out” text banner — replace with a pin icon and tooltip on hover
  • Window title bar: 28–32px maximum
  • Minimap, ARA chat, and Contents panel must not all stack on the right side simultaneously — they need assigned, non-conflicting positions
  • Tabs must be visible when windows are in fullscreen mode, not only in split-screen
2.8 — Text and Icon Size
Observation

Text in several widgets (File Explorer, Inbox Zero, control panels) is too small for comfortable extended use even at maximum text settings. Icons and controls that are too small or too densely packed are difficult to click reliably. Enterprise software used for multiple hours daily needs comfortable reading size and adequate spacing.

2.9 — Layout Grid Lines Visible in Normal Use
Bug

The widget layout grid renders its cell borders as permanent blue lines across the interface. These should only appear when the user is explicitly in layout-editing mode. In normal use the grid should be invisible.

2.10 — Tab Drag and Window Management
UX

Three behaviors should work simultaneously: (1) drag by tab header to reorder or move between panels; (2) drag by widget title bar to move the container; (3) pull a tab completely out to detach as a floating window, drag back in to re-dock. These should feel seamless, like a browser.

03UI Design System — Target Aesthetic
🌟
The HTML reports produced for this project use a design system that represents exactly the visual direction being requested. Those reports look significantly better than the current application because they use an intentional, CSS variable-driven system with specific card effects, typography, and spacing. Provide that design system document to the implementation team as the primary visual reference.

Reference: Combined HTML Report Design System.md (attached)
CSS Variable-Driven Theming
Foundation

All colors in all components must reference CSS custom properties — never hardcoded hex values anywhere in a component's stylesheet. With this system, implementing a new theme requires only updating the variable values. Switching dark/light mode is a single data-theme attribute change on the root element.

Core tokens: --bg, --surface, --surface-2, --accent, --green, --amber, --red, --purple, --text, --muted, --ink, --border, --border-hover, --shadow-card, --spotlight. Each color also has an RGB companion for use in rgba() expressions.

Card / Widget Design
Polish
  • Cursor-tracked ambient spotlight — a soft radial gradient that follows the mouse across the card surface, fading toward edges. Pure CSS + a few lines of JavaScript (no animation library).
  • Clean 1px border using var(--border), brightening to var(--border-hover) on hover
  • Subtle lift on hover: translateY(-3px) with expanding shadow — smooth, not dramatic
  • 15px border-radius on cards; 8px on smaller elements; 999px on pills/chips
  • 200ms ease transitions on all color, border, and shadow changes
Icon System — Not Emojis
Polish

Replace Unicode emoji in UI controls, buttons, navigation, and status indicators with a proper icon library. Recommended: Lucide Icons (lightweight, consistent visual weight, MIT license, scales cleanly, CSS-colorable with theme variables).

Emoji remain appropriate for user-generated content and AI responses. They are not appropriate for navigation items, button labels, status indicators, or widget headers. The distinction is immediately visible to anyone used to professional software.

Header / Chrome Ratio
Layout

Content area should occupy at least 70% of the available height. Window chrome should be minimal:

  • Window title bar: 28–32px maximum
  • Remove the “Drag outside window to pop out” text banner entirely — replace with a pin/dock icon that shows a tooltip on hover
  • Navigation tabs: single row, 32–36px height, nothing else between nav and content
Typography
Polish
  • Base font: Inter Tight (already used in reports; matches the target aesthetic perfectly)
  • Body: 15–16px, weight 400, line-height 1.65
  • Labels and secondary: 13px, weight 500
  • Minimum readable: 11px for chips and metadata — never smaller
  • No gradient text in the application UI — gradient text is a report decoration. Application headings should use var(--ink) for readability.
Theme / Skin System
Customization

With the CSS variable system in place, custom themes require only updating the variable values. Goals:

  • Dark theme (default): deep near-black backgrounds, subtle surface elevation, neon-adjacent accents
  • Light theme: clean white surfaces, deep ink text, muted accent variants
  • “Fey” theme (aspirational): a distinct third aesthetic — earthy, warm, unique palette. The premium skin.
  • User-customizable: once the variable system is in place, custom themes should be creatable by editing token values in Settings
04File System & Workspace Organization
4.1 — Workspace Directory and iCloud Sync
Foundation

The workspace should be linked to a directory on the local filesystem and optionally iCloud Drive. Changes in either direction — creating a file in the dashboard or dropping a file into the Finder folder — should be reflected automatically. Workspace root configurable and clearly visible in the interface.

4.2 — Workspace Hierarchy (Domain → Project → Thread)
Organization

A suggested three-level hierarchy. Each Thread folder contains: brain dump file (append-only), notes file, versioned reports folder, references folder, and system/memory folder. This structure corresponds to actual folders on disk. Alternative models are open for discussion.

4.3 — File Explorer
Navigation

A comprehensive file manager in the style of AntiGravity × Obsidian × macOS Finder.

Open Files / Tab Bar
  • Open files shown as tabs at the top; individually closeable; "Close All" button; download and split buttons in header
Navigation
  • Click expand arrow to expand tree inline; double-click folder name to navigate into it (Finder-style); back button, home button, right-click → "Show in Finder"
  • Quick jump to any Domain/Project/Thread; hierarchy scoping to lock a pane to the active Thread when in Scribe
Sort, Display and File Operations
  • Toggle alphabetical or last-modified sort; folders first; file type badges (.md, .pdf, .txt, .json, .docx); expand/collapse all
  • New File / New Folder (inline rename), Rename, Delete (confirmation), Copy Path, Show in Finder
  • Move to Thread (modal); Re-ingest; Convert to Markdown (PDF, DOCX, XLSX; OCR for images); batch convert on multi-select
Drag and Drop and Split Panes
  • Between panes/folders (move; Alt = copy); into Scribe editor (reference link); into agent chat (attaches as context); Cmd+V in pane pastes clipboard image as PNG
  • Up to three independent resizable panes; "Open in Split" right-click on any folder; thread switcher footer with expandable drawer
4.4 — Domain Isolation and Scoped Views
Architecture

Domains, Projects, and Threads should be relatively self-contained knowledge spaces. Cross-Domain connections should be deliberate. Domain/Project/Thread names should be excluded from the tag vocabulary to prevent structural labels from creating spurious cross-Domain graph connections via common word matching.

05Scribe — Document Editor
5.1 — Core Editor
Foundation
  • CodeMirror 6 with markdown syntax highlighting
  • Inline markdown rendering (Obsidian-style) — formatting renders in the editor; raw syntax visible when cursor is on the line, formatted output shown when cursor moves away. No separate preview pane.
  • Auto-save. Document versioning — saving a Report creates a new numbered version (v1, v2, v3…). Brain dumps and notes are append-only.
  • Contents panel — persistent, left side, not a disappearing dropdown
5.2 — Brain Dump / Intake Tab
Intake

A sticky, always-visible "Dump" tab. Clicking it switches to a full-height dump intake area. Submitting auto-prepends a # Prompt N header, saves to the brain dump file, sends to the agent. Chat shows a compact link not raw text. "Report" button appears after first dump. Not currently in the Dwellium build.

5.3 — Right-Click Menu and Paste Variants
Editing
  • Cut, Copy, Paste (verbatim)
  • Paste+ (Cmd+Shift+V) — smart paste respecting markdown structure; merges soft-wrapped prose, preserves code fences, respects lists and blockquotes
  • Paste++ (Cmd+Shift+Alt+V) — raw paste stripping all whitespace to flat string
  • Markdown submenu — Bold, Italic, Strikethrough, H1–H3, Bullet List, Numbered List, Blockquote, Code Block; Clear Formatting
5.4 — Citation Behavior
AI

Selecting text and triggering a citation (Cmd+L) creates a citation pill in the chat input area. The pill appears ready for the user to compose their own message — it does not auto-send or ask the agent anything. Distinct from inline comments (which stay in the document margin).

5.5 — AI Redlines, Inline Comments, Versioning
AI
  • Redlines — agent-proposed edits displayed as tracked changes; accept/reject per change or accept all; works for batches
  • Inline comments — margin-anchored, stored in sidecar JSON file alongside the document
  • Scribe theme customization — color picker for every syntax element (H1–H4, Bold, Italic, Code, Blockquote, Links, Frontmatter); auto-fork on preset edit; save/delete custom themes; live preview; independent from app theme
5.6 — Panel Layout, Sticky Chat, Additional Features
Layout
  • Panel docking — file explorer (Cmd+Shift+1), chat (Cmd+Shift+3); resizable dividers; chat panel must never float over the document
  • Sticky user message — original message stays pinned at top of viewport when scrolling through long agent response (position: sticky)
  • Toolbar — Undo/Redo (live history state), Bold/Italic/Strikethrough, H1–H3, lists, Blockquote, Code Block, flag markers, Focus Mode, word count. Requires history boundary fix before shipping undo/redo.
  • Find and Replace — Cmd+F / Cmd+H / Escape
  • Focus Mode — Cmd+Shift+F; sidebar and chat hidden; editor expands full width
  • Inline flags — ::flag::, ::todo::, ::question:: with colored indicators; indexed in rag_flags; searchable via Codex; HUD shows counts per Domain
  • Document priority — high/normal/low badge in tab header and Codex list
06Agent Context — Thread Assignment & Memory
6.1 — Thread Assignment (Home Directory)
Architecture

The agent is always "homed" to the active Thread — all writes go there. The active Thread label is always displayed at the top of the workspace. The file explorer can navigate freely throughout the workspace regardless of which Thread is home.

6.2 — Cross-Thread Document Access
Architecture
  • Drag to chat (read-only): gives the agent read access without moving or re-tagging the document
  • Cross-Thread edits: create a cross-thread association; document surfaces in both threads’ knowledge graph
  • Share vs. copy: share (one file, edits reflected everywhere) or copy (independent duplicate) when moving documents between Threads
6.3 — Chat History and Compaction
Memory

"Clear conversation" compresses rather than deletes. Summarization extracts key facts and outputs into a human-readable markdown Memory file in System/Memory/ (openable in Scribe). Future sessions load the compressed Memory as context. Raw history preserved in database.

07Knowledge & Intelligence Layer
7.1 — RAG Ingestion and Three-Tier Wiki
Foundation
  • Auto-ingest on file create/change. AI tag extraction. Namespace-aware. Idempotent. Cost-logged. Tag-overlap graph edges.
  • Namespace isolation: Domain/Project/Thread names excluded from tag vocabulary to prevent common-word spurious connections.
  • Three-tier wiki: auto-generated synthesis pages at Thread, Project, and Domain levels via Claude Sonnet on each ingest; cold-start bootstrap on first launch.
7.2 — Synthesis, Foundry, and Knowledge Graph
Intelligence
  • Synthesis compounding loop: Ingest → Compile → Query & Synthesize → Capture → Return → Recompile. Synthesized outputs feed back into the corpus. "Second-layer query" as a named one-click operation.
  • Foundry intake: Capture (URL, paste, file, iCloud) → AI Triage (proposes tags, location, quality) → Review → Admit. Tracks separately from ingested documents.
  • Knowledge graph: d3-force visualization; betweenness centrality; Louvain community detection (graphology + graphology-communities-louvain); structural gap detection; double-click to open in Scribe.
  • MCP exposure: if accessible via MCP, the knowledge base is queryable from any Claude session (rag_ingest, rag_query, rag_compile).
08Agent Management — The Hive
8.1 — Agent Control Room
Agents

Central panel: agent status cards (idle/running/error, last run, recent output), cost attribution by agent and Domain, manual trigger for any agent, per-agent configuration, Dreams panel surfacing Honcho’s derived insights.

8.2 — CoPaw Auto-Capture
Memory

After every agent response, silently extract key facts and write to the memory system. Approximately 50 lines of code. The mechanism that makes memory compound continuously from the flow of ordinary work rather than only from explicit saves.

8.3 — Development Tools
Tools
  • Schema Producer — given a data structure description, produces a structured schema
  • PRD synthesis agent — multiple source documents synthesized into structured requirements
  • Gap analysis agent — given a specification and implementation, identifies what is missing or inconsistent

10Goal
What Success Looks Like

A dashboard that can be used comfortably for hours of daily work as the primary working environment for developing AstraStrata. The better this build is, the faster the PRD work can proceed — which is the critical path to completing the full project.

Success means: the system is installed and pleasant to use; files are organized and findable; the Scribe editor is a capable writing environment; the knowledge system is ingesting content and surfacing useful synthesis; agents can be configured and directed; the overall UI feels like it was designed by people who actually use it; and the remaining feature gaps described in Sections 4–8 of this document are addressed in the next development cycle.

✔
The features described in Sections 4–8 are recommendations and starting points. The implementation team is encouraged to improve upon them. The goal is a system that exceeds what these notes describe, not one that merely meets them.
