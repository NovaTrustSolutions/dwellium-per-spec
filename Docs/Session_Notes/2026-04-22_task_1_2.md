## The 2-STORY UUID

## Data-collision resolution

The Task 1.2 briefing assumed no 2-STORY entity existed in `entities.json` and instructed adding a new one with a fresh deterministic UUID. Inspection showed an entity with the exact name was already present.

**Decision:** extend the existing entity with the typed blocks rather than add a duplicate.

- Adding a duplicate would fork downstream vendor lookups (two same-named vendors).
- The contract test binds to the one canonical UUID instead of needing disambiguation.
- Randy's Tub-N-Tile at UUID ending `…2716` (coincidental collision with AppFolio's `vendor_id=2716`) is unchanged.

## Trade-offs & deviations from plan

1. **No `chore(types): @deprecated` commit.** The plan asked for JSDoc `@deprecated` on `coiStatus` / `coiExpiry` / `w9OnFile` / `insuranceCarrier`. Those fields are not first-class members of `EntityProfile` — they live under `metadata: Record<string, any>` (an untyped bag). JSDoc cannot target dictionary keys, so the annotation has no surface to attach to. Documented in the `feat(types)` commit body. The 2026-Q3 cleanup will lift the metadata strings into `vendorCompliance` and delete the bag keys.
2. **Tax ID mask further-redacted.** Plan specified `taxIdMasked: 'XX-XXX4232'` from the reference JSON. That value matches the PII scanner's `ssn-mask-leak` pattern (`/\bXX-XXX\d{4}\b/`) and would fail CI. Redacted to `XX-XXX-XXXX` (allowlisted). Contract test asserts presence, not the literal last-4.
3. **Legacy metadata `paymentType`** on the 2-STORY entry was `"Check"` pre-Task-1.2 — updated to `"Zelle"` to match the canonical `paymentMethod` field and Lisa's 1/17/2024 AppFolio note. Legacy consumers reading `metadata.paymentType` now see the correct channel; typed consumers use the canonical field.
4. **Test file renamed `.ts` → `.tsx`.** The contract test uses JSX-bearing `render()` calls (testing-library); vitest config includes `tsx` so no config change needed.
5. **Manual dev smoke deferred.** Same Vite env-replacement quirk observed on Task 1.1. Contract tests cover the render seam (ComplianceTab + AccountingTab against the 2-STORY seed) which is the material risk.
